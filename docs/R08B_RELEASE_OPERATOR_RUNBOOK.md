# R08B bounded release operator runbook

This runbook is a frozen input to the V2.4 fast working release. It describes
only local, disposable Linux and agent-created isolated execution. It grants
no deployment, restore, reboot, publication, canonical/domain mutation, live
trading or autonomous live-security authority. A real live/VPS host and the
24-hour, 48-hour, seven-day and 14-day windows are out of scope.

## Release identity

The release subject does not exist until the R08B parent PR is squash-merged
and the exact merge-head CI is green. Before that event all candidate fields
are `null`/`NOT_CUT`. After it, the immutable R08B candidate-cut receipt is the
only source for:

- `candidate_release_sha` and `candidate_tree_sha`;
- exact merge-head CI run;
- `linux/amd64` image ID and owner-only archive SHA-256;
- SPDX SBOM and package-inventory SHA-256 values;
- dependency lock, notices, config, policy and provider fingerprints;
- release-relevant-path manifest and evidence-only path allowlist SHA-256.

Never substitute a branch name, tag, prior S38 candidate, locally rebuilt
image, registry tag or CI run for those immutable values.

## Universal stop conditions

Stop the current disposable action, preserve its evidence and issue no retry
when any of these conditions occurs:

- any SHA, tree, CI, image, archive, SBOM, inventory, config, policy, provider
  or release-path fingerprint differs from the accepted R08B/F09 receipt;
- a path outside the frozen evidence-only allowlist changed after R08B;
- a target has a repository/worktree mount, prior runtime volume or
  pre-rendered state;
- the target is live, production, VPS, shared, user-owned or not disposable;
- a private credential or raw response would enter Git or command output;
- the Core/L0 side has provider network, more than one durable writer exists,
  or a connected role lacks its exact principal binding;
- budget is absent/exceeded, an ambiguous provider action is not durably
  `UNKNOWN`, or an automatic retry would occur;
- a command would perform canonical/domain mutation, publication, trading,
  security execution, live restore/rollback/reboot or authority escalation.

Failures are evidence, not permission to weaken a check or use a different
release.

## R08B candidate verification

Use an exact detached checkout of the accepted `origin/main` candidate. Keep
all generated bytes in a new owner-only directory outside the repository.

```text
git rev-parse HEAD
git rev-parse refs/remotes/origin/main
git rev-parse HEAD^{tree}
gh run view <candidate-ci-run> --json headSha,status,conclusion,event
python3 tools/candidate_image_build.py verify \
  --candidate-sha <candidate-release-sha> \
  --candidate-ci-run <candidate-ci-run> \
  --output-dir <owner-only-candidate-directory>
python3 ops/release/image_e2e_harness.py verify-inputs
python3 tools/verify_final_release_freeze.py verify-v2.4-status
```

Both Git SHAs must equal the receipt, CI must be completed/success, and every
tool must pass. The candidate archive must be mode `0600`; its SHA and embedded
image config digest must equal the receipt. Do not rebuild to repair a failed
identity check.

## R08C exact-image functional qualification

R08C is `LOCAL_ONLY`. Load only the accepted archive into an isolated local
container engine, select the immutable image ID (never a mutable tag), and run
the shipped exact-image harness through the real AF_UNIX boundary. Reuse the
R08B private real-source and role bindings; do not make a replacement provider
call merely to refresh metadata.

The accepted path must be non-vacuous:

```text
real D0/D1 collector
→ Worker proposal
→ separate Critic decision
→ draft
→ ADMIT
→ Job / Permit / Lease
→ offline Core/L0 execution
→ positive, negative or inconclusive validation
→ durable feedback and memory
→ next event
→ WAIT_AUTHORITY
```

Run the frozen hostile matrix for empty/mixed policy, role spoof, forged or
expired authority, duplicates, second writer, crash boundaries, budget,
malformed model output, `UNKNOWN` without retry, `AI_OFF`, private/live request
and release-fingerprint substitution. Core/L0 must remain network-none; only
the connected edge may use the bounded provider route. Seal private evidence
and publish only sanitized digests. No Git mutation occurs in R08C.

## F09 qualification freeze

F09 may add only authority and sanitized evidence under the R08B evidence-only
allowlist. Recompute the release-relevant-path manifest against the current
evidence head and prove every non-allowlisted blob still equals R08B. Bind the
R08C typed receipt, current capability proofs, exact image/config/policy/
provider/SBOM/inventory values, this runbook and the deployment/rollback packet.

The F09 state is exactly
`FROZEN_CANDIDATE_READY_FOR_DISPOSABLE_QUALIFICATION`. It is not
`RELEASE_READY`, grants no authority and does not change the R08B release
subject SHA/tree. Only `evidence_head_sha` advances.

## F10 clean disposable Linux install

Create a fresh Linux target owned by the agent. Before installation prove and
receipt all of the following:

- target identity is disposable and is not a live/VPS host;
- no repository/worktree path is mounted;
- no prior runtime volume, database, config or service exists;
- the only release bytes are the verified F09 archive/packet bundle;
- secret access uses an existing owner-only locator, never an inline value;
- intended non-root/rootless UIDs and systemd user supervision are available.

Install only the shipped image/archive, config template, systemd units,
monitor/backup policies and controllers. Render private config outside Git;
the rendered config must preserve frozen public fingerprints and the R08B
collector/Worker/Critic bindings. Enable and start the core, connected worker,
monitor and backup timer topology under one supervisor and one durable writer.

Prove startup, autostart, monitor causality, a same-release restart, the entire
R08C AF_UNIX cycle and the hostile matrix. Then use
`tools/release_backup_restore.py backup` through the shipped backup unit to
create an encrypted verified backup of the actual post-JobSpec state. Seal its
exact digest, manifest, release fingerprint and action receipt for F11. No
other target or Git state may be used as evidence.

## F11 isolated restore and rollback campaign

Use the exact F10 backup sequentially on two distinct new clean isolated
targets. Each restore requires its own action ID and owner-only receipt. Prove
manifest equality, one-writer identity, no duplicate completed work, safe
pending-work recovery, post-restore AF_UNIX functionality and exact R08B/F09
release fingerprints.

On isolated targets only, exercise provider outage, missing credential,
timeout/`UNKNOWN`, `AI_OFF`, crash/idempotency/fencing and same-release service
restart. Ambiguous calls are reconciled once and never automatically retried.

The rollback action first applies a clearly distinct, non-releasable
fault-injection fingerprint to a separate isolated target. Only then restore
the exact frozen F09 release and complete a post-rollback JobSpec. A
same-version reinstall is not rollback evidence. The historical superseded
candidate is forbidden. Seal the campaign before deleting agent-created
targets; target deletion is cleanup, not evidence.

## F12 closeout

F12-A integrates only sanitized F10/F11 identities and receipts, proves the
full evidence DAG and seals an immutable terminal receipt head with
`PENDING_FINAL_AUDIT`. Repository status remains `IN_PROGRESS`; no `DONE`
claim is permitted. After exact receipt-head CI, plan-owned delivery artifacts
are cleaned only after proving they are accepted and contain no unique WIP.

F12-B is a separate Agent-5 read-only audit outside Git against exactly the
F12-A receipt head. It records critical/high findings equal to zero, its own
SHA-256, owner-only recoverable locator and retention policy in external
`EXECUTION_STATE`. No Git commit follows F12-B.

Only that atomic external transition may set `PRODUCT_DONE`, `RELEASE_DONE`,
`REAL_BOUNDED_RESEARCH_OPERATION_READY` and `MASTER_PLAN_DONE` true with
`status=DONE`. It must retain:

```text
PHYSICALLY_DEPLOYED=false
OPERATIONALLY_PROVEN=false
TIMED_WINDOWS=OUT_OF_SCOPE
LIVE_VPS_DEPLOYMENT=OUT_OF_SCOPE
```

## Cleanup and retention

Clean only plan-owned disposable containers, volumes, branches and worktrees,
and only after immutable receipt sealing plus read-only proof that no unique
unpushed work exists. Never delete the accepted image archive, F10 backup or
private receipts before their declared retention point. Never prune user or
foreign WIP. Record cleanup identities and outcomes; absence of a temporary
directory after reboot is not acceptance evidence.
